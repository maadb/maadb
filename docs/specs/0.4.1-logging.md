# 0.4.1 — Structured Logging + Audit Channel

Spec for structured JSON logging of every MCP tool call, plus a dedicated audit channel for write operations. Replaces the existing console-only text logger as the default output while preserving the internal `logger` primitive.

## Problem

Today `src/engine/logger.ts` emits plain-text lines to stderr via a pluggable `LogHandler`. That's enough for local development but insufficient for:

1. **Production debugging.** When a hosted deployment has an issue, grepping a text stream for what a single request did is error-prone. Structured JSON with a `request_id` lets an operator trace one request end-to-end.
2. **Compliance / audit.** A write-only channel with before/after state is what any operator eventually asks for. Mixing audit events with operational noise makes both harder to consume.
3. **Latency / error observability.** Without per-request `latency_ms` and `error_code` fields, operators can't spot slow tools or rising error rates without parsing free-form text.

## Goals

- Every MCP tool call produces exactly one structured log line, JSON, to stdout (or configurable sink).
- Every successful write emits an additional line on a dedicated audit channel, including before/after frontmatter version.
- Sensitive values (bearer tokens) never appear in logs.
- Existing engine `logger` primitive continues to work — its handler is swapped to a JSON emitter.
- Log level controlled by env.

## Non-goals

- OpenTelemetry / distributed tracing. Deferred to 0.8.5. `request_id` plus structured logs cover the MVP tracing need.
- Metrics endpoint (Prometheus scrape, OTEL push). Deferred to 0.8.5.
- Async log shipping (syslog, fluentbit). Operators can pipe stdout to whatever they want.
- Rotation, archival. Handled downstream by whatever consumes stdout.

## Design

### Library choice: pino

Adding `pino` as a production dependency. Reasons:
- Fastest JSON logger in the Node ecosystem; near-zero overhead for disabled levels.
- Synchronous output option for tests and shutdown safety.
- First-class redaction of sensitive fields via path spec.
- Widely understood — ops teams know how to consume it.

Alternative considered: hand-rolled JSON emitter. Rejected — reimplementing pino's redaction, level gating, and sync-mode support is a week of work for no upside.

### Two loggers, one library

```ts
// src/logging.ts (new)
import pino from 'pino';

export const opsLog = pino({
  level: process.env.MAAD_LOG_LEVEL ?? 'info',
  redact: {
    paths: ['args.authorization', 'args.token', 'args.headers.authorization'],
    censor: '[redacted]',
  },
  // ops log goes to stdout by default; operators redirect as needed
});

export const auditLog = pino({
  level: 'info',                  // audit is always on
  base: { channel: 'audit' },
  // audit log goes to a separate file handle if MAAD_AUDIT_PATH is set,
  // else falls through to stdout with channel='audit' field for easy grep
});
```

Two logger instances, same library. Ops log carries operational chatter; audit log carries a curated stream of state-changing events.

### Ops log: one line per MCP call

Location: inside `withSession` / `withEngine` wrappers in `src/mcp/with-session.ts`. The wrapper brackets each tool call:

```ts
const requestId = randomUUID();
const startMs = Date.now();
// ... session / role / rate-limit checks ...
const response = await toolHandler();
const latencyMs = Date.now() - startMs;

opsLog.info({
  event: 'tool_call',
  request_id: requestId,
  session_id: sessionId,
  project: activeProject,
  tool: toolName,
  role: effectiveRole,
  payload_size: JSON.stringify(args).length,
  latency_ms: latencyMs,
  result: response.ok ? 'ok' : 'error',
  error_code: response.ok ? null : response.errors?.[0]?.code ?? null,
}, 'tool_call');
```

Every field on every call. Missing values are explicit `null`, not omitted, so downstream parsers have a stable schema.

Optional: `request_id` is attached to `response` as `_meta.request_id` so clients can quote it in bug reports. Non-breaking (MCP allows extra fields).

### Audit log: one line per successful write

Fires from the MCP write-tool handlers, after the engine call succeeds:

```ts
auditLog.info({
  event: 'write',
  request_id: requestId,
  session_id: sessionId,
  project: activeProject,
  tool: 'maad_create' | 'maad_update' | 'maad_delete' | 'maad_bulk_create' | 'maad_bulk_update',
  doc_id: result.docId,                // for singular writes
  doc_ids: result.succeeded.map(...),  // for bulk
  doc_type: ...,
  version_before: ...,                 // null on create; prior version on update/delete
  version_after: ...,                  // new version after write
  changed_fields: [...],               // from engine result; empty array on create
  git_commit: ...,                     // commit SHA from git layer, if available
}, 'write');
```

Writes that failed (validation, duplicate id, etc.) do not emit an audit line. They're visible in the ops log with `result: error`. Audit is strictly "what actually changed."

### Redaction

Pino's `redact` path config strips the `authorization` header and common token-like fields. At the MCP tool-args level, the only commonly sensitive fields are:
- None by current tool surface (MAADB writes don't carry auth material).
- The remote-MCP layer (0.5.0) will add bearer tokens in transport headers — those are redacted at the pino config level above, not at the tool layer.

Explicit rule: **tool args themselves are logged verbatim.** If a future tool accepts a sensitive value in its body, that tool must add its field to the `redact.paths` list at registration time. Document this in the tool-author guide.

### Env knobs

| Var | Default | Meaning |
|---|---|---|
| `MAAD_LOG_LEVEL` | `info` | pino level for ops log (trace / debug / info / warn / error / fatal / silent) |
| `MAAD_LOG_PRETTY` | unset | if set, ops log uses pino-pretty transport (for local dev) |
| `MAAD_AUDIT_PATH` | unset | if set, audit log writes to this file instead of stdout |

### Engine logger migration

Swap `src/engine/logger.ts`'s `defaultHandler` from `console.error` to a call through `opsLog`:

```ts
const defaultHandler: LogHandler = (entry) => {
  opsLog[severityToPinoLevel(entry.severity)](
    { event: 'engine', category: entry.category, operation: entry.operation, ...entry.details },
    entry.message,
  );
};
```

Existing callers continue to use `logger.bestEffort(...)` etc.; output flips from plain text to JSON automatically. No caller changes required.

## Test plan

`test/mcp/logging.test.ts`:

### L1 — Every tool call produces one ops log line
Capture ops log via injected pino destination. Call `maad_summary`. Assert exactly one log line with `event: 'tool_call'`, all expected fields present.

### L2 — Successful write produces one audit line
Call `maad_create`. Assert one audit line with `event: 'write'`, `version_before: null`, `version_after: 1`, `git_commit` populated.

### L3 — Failed write produces ops line only
Call `maad_create` with invalid fields. Assert one ops log line with `result: 'error'` and `error_code: 'VALIDATION_FAILED'`, zero audit lines.

### L4 — Update emits version_before and version_after
Create a doc, update it. Audit line for the update has `version_before: 1`, `version_after: 2`, `changed_fields: [...]`.

### L5 — Redaction
Call a tool with args containing an `authorization` field. Assert the logged args show `[redacted]` not the original value.

### L6 — Level gating
Set `MAAD_LOG_LEVEL=warn`, run a normal call. Assert no ops log line (info gated out). Assert an error call still logs.

### L7 — Audit path separation
Set `MAAD_AUDIT_PATH=/tmp/audit.log`, write a doc. Assert ops log doesn't contain the write event; audit file does.

### L8 — Request ID propagation
Call a tool, capture the response. Assert `_meta.request_id` in response equals the `request_id` in the matching ops log line.

## Interactions

- **Mutex (H2):** mutex events (acquire, release, slow write) emit via ops log through the engine logger handler. No direct change here.
- **Idempotency (H4):** replays emit `event: 'idempotency_replay'` on the ops log. No audit line — the write didn't actually happen again.
- **Rate limit (H5):** rejections emit `event: 'rate_limited'` on the ops log with the rejection reason.
- **Graceful shutdown (H7):** ops log emits `event: 'shutdown_start'` / `shutdown_complete` around the drain; audit log closes cleanly on shutdown so the last write is flushed.
